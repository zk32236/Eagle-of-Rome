## 四、6 阶段路线图确认

顾问确认后续 GUI 改造采用以下 6 阶段路线图：

| Phase | 名称 | 核心目标 | 当前状态 |
| --- | --- | --- | --- |
| Phase 1 | Shell + 天命 Vertical Slice | 重建全局外壳，并验证天命阶段完整闭环 | 下一步准备执行 |
| Phase 2 | 人口视觉重建 + 收入数据层 | 人口阶段视觉对齐；收入阶段补 GUI API / DTO / Store / QML | 后续 |
| Phase 3 | 广场阶段 Forum 重建 | 接入 forum API，重建广场阶段玩家交互 | 后续 |
| Phase 4 | 元老院交互化 | 在只读基础上接入提案、投票、否决、结算 | 后续 |
| Phase 5 | 战争 + 决算 | 新建 GUI 安全 combat / resolution API，接入战争与决算 | 后续 |
| Phase 6 | 收尾 + Deferred 项 | 处理占位查询、视觉 polish、i18n、多分辨率、未决产品项 | 后续 |

### 4.1 Phase 1 目标

Phase 1 只做：

- 顶栏视觉对齐。
- 左侧 7 阶段导航视觉重建。
- 右侧状态 / 日志栏视觉对齐。
- 底部查询栏视觉对齐。
- Modal / Overlay 视觉对齐。
- 天命阶段视觉重建与完整动作闭环。

Phase 1 的目标不是完成完整 GUI，而是证明：

```text
QML
↓
GuiSessionStore
↓
GuiApiAdapter
↓
API
↓
Core / System / Service
↓
Entity
```

这一链路在新 QML 视觉体系下可以稳定运行。

### 4.2 Phase 2 目标

Phase 2 拆成两条可并行子线：

- Phase 2a：人口阶段视觉重建。
- Phase 2b：收入阶段数据层构筑。

收入阶段必须注意：`EconomicService` 已存在，但缺 GUI 安全 API、DTO、Store 暴露和 QML 页面。不得把收入阶段误写成“后端规则从零缺失”。

### 4.3 Phase 3 目标

广场阶段已有 `forum_api.py` 的较多后端能力，后续重点是：

- Adapter 暴露。
- Store 属性与 Slot。
- `ForumStage.qml`。
- 解雇、人才市场、竞标、公地、凯旋、公示结算等子区域。

广场子环节状态机属于 Phase 3 或更后续，不得提前塞入 Phase 1。

### 4.4 Phase 4 目标

元老院阶段当前只读 DTO 链路较完整，但交互能力未接 GUI：

- 提案创建。
- 投票。
- 保民官否决。
- 元老院结算。

该阶段需要单独审查政治交互边界。

### 4.5 Phase 5 目标

战争与决算不能直接复用 CLI Command，必须先建立 GUI 安全 API：

- `combat_api.py`
- `resolution_api.py`
- 稳定 DTO。
- Adapter / Store 暴露。
- `CombatStage.qml`
- `ResolutionStage.qml`

### 4.6 Phase 6 目标

Phase 6 用于处理：

- 底部 Deferred 查询按钮。
- 视觉 polish。
- i18n 收束。
- 多分辨率验收。
- 稳定度等未决产品项。
- 是否需要统一 Store 日志。
- 是否需要通用 `advancePhase`。

注意：Phase 6 中的未决项不得在当前文档中写成“最终必做方案”。
