# OC GUI v3.25.1 重建可行性审计指令

任务编号：GUI-P0-REBUILD-AUDIT-2026-07-10  
任务名称：EOR GUI v3.25.1 重建可行性审计  
任务类型：架构审计 / GUI 改造决策前置任务  
优先级：P0  
目标版本：MVP 0.9 GUI 重建准备阶段  
任务发起：GUI 开发顾问 / Codex  
执行对象：OC  

## 一、背景说明

EOR 当前已经存在一套 GUI 实现，但前期开发缺少稳定界面设计原型，存在“直接写代码、边写边调整”的历史问题，导致后续视觉、布局、交互与功能调整成本较高。

当前 GUI 产品目标已经明确：

- `EOR_GUI_Prototype_v3.25.1.html` 是后续 GUI 的正式目标形态。
- `EOR_GUI_Design_Guidelines_Codex_v4.0_v3.25.1_baseline.md` 是 GUI 视觉、层级、组件表现与验收标准的设计基线。
- 本次讨论的“推倒重建”仅限 GUI 实现层，不包括 Core、API、游戏规则和后端系统重写。

本任务的目的不是立即开始重写 GUI，而是先由 OC 提供足够准确的代码现状信息，使顾问能够判断后续应采用哪种改造路线。

## 二、任务目标

请 OC 基于当前 EOR 项目代码，完成一次短周期、可回溯的 GUI 实现层审计，回答以下核心问题：

1. 现有 GUI 框架哪些部分可以保留？
2. 哪些部分应当废弃或重建？
3. 若以 v3.25.1 HTML 为目标实现正式 GUI，继续修补旧 GUI 是否比受控重建成本更高？
4. 推荐采用以下哪一种路线：
   - A：保留现有 GUI 框架，仅做局部改造。
   - B：保留启动入口、Adapter、Store 等连接层，重建 QML 页面与组件。
   - C：GUI 层整体重建，仅复用 API、DTO、测试入口和必要工程配置。

请注意：OC 本轮只做审计和报告，不修改代码。

## 七、推荐路线判断

请在报告中明确选择 A / B / C 之一，并给出理由。

| 选项 | 含义 | 适用条件 |
| --- | --- | --- |
| A | 保留现有 GUI 框架，仅做局部改造 | 现有结构清晰，布局和状态管理接近 v3.25.1 |

请避免模糊结论。若存在不确定项，请列为“决策阻塞信息”，而不是给出折中建议。
