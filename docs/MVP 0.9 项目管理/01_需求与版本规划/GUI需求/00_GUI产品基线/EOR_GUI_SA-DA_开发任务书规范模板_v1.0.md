适用范围：EOR GUI v3.25.1 HTML 目标形态在 QML/PySide6 项目环境中的实现  
正式存档目录：`E:\OpenClaw\Projects\EOR\agents\SA\GUI\`（唯一可信源）

## 一、制定目的

当前 GUI Phase 1 实测显示，DA 能理解 v3.25.1 的大致风格，但容易出现结构级偏差，例如：

- 中央区域被实现成深色背景加大块羊皮纸卡片，而目标稿是一个完整的象牙白中央桌面。
- 右侧面板顺序被改成资源卡片优先，而目标稿是“当前阶段、操作、进度、玩家、事件日志”。
- 左侧阶段栏过宽、按钮过大，偏离目标稿的窄轨道图标按钮形态。
- 底部查询栏被实现成状态点按钮，而目标稿是图标加文字的查询入口。
- 主操作按钮位置偏左，目标稿要求位于中央桌面底部居中附近。

因此，后续 SA 给 DA 的 GUI 任务书必须把“视觉结构复刻”写成可执行、可检查、可返修的约束，而不是只给 HTML 原型和笼统描述。

## 二、核心原则

1. 先复刻结构，再绑定功能。
   GUI 任务不得一边自由调整布局，一边接功能。每个阶段应先完成静态结构对齐，再接 Store/API。

2. 设计稿是布局约束，不只是风格参考。
   DA 不能只复用颜色、边框、字体氛围后自行重排区域。区域位置、层级、主次关系必须与目标截图一致。

3. 每个 GUI 任务必须提交截图自检。
   没有运行截图、没有区域对照、没有差异说明的 GUI 交付，不应进入验收。

4. QML 只做表现和状态绑定。
   不得在 QML 中复制 Core、System、Service 或命令层业务规则，不得绕过 Adapter/Store。

## 三、每份 GUI DA 开发任务书必须包含的字段

| 字段 | 必填要求 |
| --- | --- |
| 任务编号 | 使用 GUI-P0-xx 或当前项目编号体系，明确 Phase 和轮次 |
| 任务目标 | 说明本次要复刻的页面、阶段、交互闭环 |
| 目标截图 | 必须给出目标截图路径，优先使用 v3.25.1 设计图 |
| 固定验收视口 | 至少包含 1440x900，必要时补充 1366x768、1280x720 |
| 参考原型 | 指向 `EOR_GUI_Prototype_v3.25.1.html` 或对应 HTML 文件 |
| 设计规范 | 指向 `EOR_GUI_Design_Guidelines_Codex_v4.0_v3.25.1_baseline.md` |
| 补充规范 | 指向 `EOR_GUI_QML_Rebuild_Supplement_v3.25.1.md` |
| 区域编号 | 将界面拆成 A 顶栏、B 左栏、C 中央桌面、D 右栏、E 底栏、F 主操作 |
| 允许修改范围 | 明确可修改 QML、Store、Adapter、测试文件 |
| 禁止修改范围 | 明确不得修改 Core/System/Service/Entity，除非任务特别授权 |
| 状态绑定表 | 每个按钮、显示字段、禁用原因都要说明绑定来源 |
| 视觉不可偏移清单 | 写出本任务不能偏离的布局、颜色、材质、位置、顺序 |
| 截图自查表 | 要求 DA 按区域提交“目标/实际/差异/是否需返修” |
| 测试要求 | GUI 启动、相关单测、回归测试，禁止硬编码历史通过数量 |
| 验收标准 | 功能、视觉、架构、测试、交付物全部可检查 |
| 返修条件 | 写明哪些偏差出现时直接退回，不进入细节验收 |

## 四、固定验收视口

每个 GUI 任务书必须指定至少一个主验收视口：

- 主视口：1440x900
- 补充视口：1366x768
- 最小桌面视口：1280x720

DA 交付时至少提交主视口截图。涉及响应式或边界布局时，必须补充 1366x768 与 1280x720 截图。

截图命名建议：

- `GUI-P0-xx_PhaseN_target_1440x900.png`
- `GUI-P0-xx_PhaseN_actual_1440x900.png`
- `GUI-P0-xx_PhaseN_actual_1366x768.png`

## 五、GUI 区域编号标准

SA 在任务书中必须使用固定区域编号，避免 DA 对“页面整体”自由解释。

| 区域 | 名称 | 说明 |
| --- | --- | --- |
| A | 顶部状态栏 | 品牌、国库、派系、影响力、稳定度、战争数、年份回合 |
| B | 左侧阶段导航 | 七阶段入口，窄轨道，图标加短文字，当前阶段高亮 |
| C | 中央阶段桌面 | 当前阶段主内容区域，v3.25.1 中应是完整象牙白桌面 |
| D | 右侧状态面板 | 当前阶段、操作、进度、玩家、事件日志 |
| E | 底部查询栏 | 查询入口，图标加文字按钮，不是状态点列表 |
| F | 主操作控件 | 当前阶段的主按钮或推进按钮，位置需服从目标稿 |

## 六、Phase 1 视觉不可偏移清单

后续 Phase 1 或同类 Shell 任务书必须明确写入以下约束：

1. 中央区域必须是一个完整的象牙白阶段桌面。
   不得实现为深色背景加一个巨大羊皮纸结果卡片。

2. 羊皮纸卡片只能用于提示、流程说明、结果记录等局部内容。
   不得把整个中央主工作区替换为羊皮纸卡片。

3. 右侧面板顺序必须是：
   当前阶段、操作、进度、玩家、事件日志。
   不得用派系资源卡片替代右栏顶部结构。

4. 左侧阶段栏必须保持窄轨道。
   不得改成宽按钮列表，也不得让按钮占用过多横向空间。

5. 底部查询栏必须是图标加文字的查询入口。
   不得使用状态圆点作为主要语义。

6. 主操作按钮在目标稿中位于中央桌面底部居中附近。
   不得默认放在左下角，除非目标截图明确如此。

7. 顶部状态栏必须保持目标稿的块数、顺序和比例。
   Phase 1 如暂缺某个数据，可以用安全占位或 Store 只读派生值，但不得擅自删除目标栏位。

## 七、架构禁止项

每份 GUI 任务书必须保留以下禁止事项：

- 不得修改 Core/System/Service/Entity，除非本任务明确授权。
- 不得在 QML 中调用 CLI command 或命令层函数。
- 不得绕过 `GuiApiAdapter` 和 `GuiSessionStore` 直接调用核心对象。
- 不得在 QML 中复制阶段业务规则、胜负规则、事件抽取规则、人口调整规则。
- 不得直接读写私有字段。
- 不得用中文显示文本作为业务逻辑判断条件。
- 不得把未接入 Store/API 的控件伪装成已完成状态。
- 不得以“大致相似”为视觉验收依据，必须按区域对照目标截图。

## 八、状态绑定表模板

SA 必须要求 DA 在任务书或交付报告中填写以下表格。

| 控件/显示项 | 区域 | Store/API 来源 | 可用条件 | 禁用原因 | 成功反馈 | 失败反馈 |
| --- | --- | --- | --- | --- | --- | --- |
| 执行天命 | F | `GuiSessionStore` / `GuiApiAdapter.doAdvanceMortality()` | 当前阶段为天命且可操作 | 非天命阶段或无会话 | 中央桌面显示天命结果，日志追加事件 | 右侧或日志显示错误消息 |
| 推进到下一阶段 | D/F | Store 当前阶段状态 | 天命已完成且允许推进 | 天命未执行或后端拒绝 | 阶段切换并更新左栏高亮 | 显示 API message/errors |
| 当前阶段 | D | Store 当前阶段 DTO | 会话存在 | 无会话 | 显示阶段名与说明 | 显示安全占位 |

实际任务中应列出本阶段所有可见按钮、关键状态字段和日志入口。

## 十、直接返修条件

出现以下任一情况，SA 或 Codex 可直接退回 DA 返修：

1. 中央区域不是完整象牙白桌面，而是深色背景加巨大卡片。
2. 右侧面板未按“当前阶段、操作、进度、玩家、事件日志”组织。
3. 左侧导航明显变宽，脱离目标稿窄轨道形态。
4. 底部查询栏使用状态点替代图标加文字入口。
5. 主操作按钮位置与目标稿明显不符。
6. 顶部状态栏缺少目标稿核心栏位，且没有说明安全占位策略。
7. DA 未提交实际运行截图。
8. DA 未提交区域自查表。
9. DA 交付中出现 Core/System/Service/Entity 非授权修改。
10. DA 用硬编码数据伪装 Store/API 已完成。

## 十一、测试要求写法

SA 不应在任务书中硬编码历史测试通过数量，例如“773 passed”。应要求 DA 报告实际运行结果：

- GUI 启动测试：说明启动命令、是否进入主界面、是否无 QML fatal error。
- GUI 单元测试：运行相关 `test_gui` 测试并报告实际结果。
- API/Adapter 相关测试：若改动 Adapter 或 Store，必须运行对应测试。
- 回归测试：运行项目当前约定回归集，并报告实际通过、失败、跳过数量。
- 手工截图验收：提交目标视口截图和区域自查表。

测试结果必须写成“命令、结果、失败项、遗留风险”，不得只写“已测试”。

## 十二、回滚与 Git 要求

SA 任务书不得默认要求 DA 使用 `git checkout --`、`git reset --hard` 等方式回滚。  
如需回滚，必须先说明回滚范围、原因，并由项目负责人或当前任务负责人确认。

如任务需要提交 Git commit，应将其写为“交付建议”或“由 SA/OC 确认后执行”，不应作为 DA 功能验收的前置条件。

## 十五、建议新增任务类型：GUI 视觉校准返修任务

当 DA 已完成初版功能但实际截图与目标稿存在结构偏差时，SA 应单独发布“GUI 视觉校准返修任务”，不要把返修混入下一阶段功能开发。

该类任务只允许修改：

- QML 布局结构
- QML 视觉参数
- QML 组件层级
- 图标、字号、间距、边框、颜色
- 必要的只读 Store 展示字段

该类任务禁止修改：

- Core/System/Service/Entity
- 阶段业务逻辑
- API 行为语义
- 新阶段功能
- 新查询功能
- 未授权数据结构

验收重点是目标截图复刻，不以新增功能为目标。

## 十六、一句话原则

SA 的 GUI 任务书必须让 DA 能够按截图逐区复刻，而不是让 DA 自由理解“v3.25.1 风格”。  
对 GUI 来说，布局结构就是需求，不是实现细节。
