# EOR GUI 设计规范 - Codex v4.0 / v3.25.1 实施基线

生成日期：2026-07-07  
更新日期：2026-07-08  
适用项目：Eagle of Rome  
关联参考原型：`EOR_GUI_Prototype_Codex_v4.0.html`  
当前实施基线原型：`EOR_GUI_Prototype_v3.25.1.html`  
定位：GUI 设计与实现规范，用于后续界面方案评审、任务拆分、GUI 验收和返工判断。

说明：`EOR_GUI_Prototype_Codex_v4.0.html` 作为视觉方向与组件质感参考；`EOR_GUI_Prototype_v3.25.1.html` 作为当前开发实施基线。后续 OC 开发应优先保持 v3.25.1 的布局、功能与交互逻辑，在此基础上持续吸收 v4.0 的视觉质量。

---

## 1. 设计目标

本规范用于定义 Eagle of Rome 后续 GUI 的界面方向、布局原则、信息密度、交互规则和验收标准。

核心目标：

1. 保留游戏既有 7 阶段流程，不改变核心规则。
2. 让玩家在一个主界面内理解“当前处于哪一阶段、我能做什么、做完后会去哪里”。
3. 将复杂策略信息组织成可扫描、可比较、可行动的结构。
4. 降低 CLI 思维残留，避免 GUI 只是命令行按钮化。
5. 保持后续 QML/真实 GUI 落地时可复用：阶段容器、状态摘要、查询弹窗、日志反馈、玩家权限提示均应成为稳定组件。

---

## 2. 总体设计方向

设计关键词：

- 共和国议事厅
- 军政沙盘
- 高信息密度
- 低装饰噪音
- 清晰阶段流
- 决策优先
- 罗马身份提示，而非重度拟物

视觉应当体现“玩家正在主持共和国事务”，而不是进入一个纯现代后台系统。界面可以有铜色、羊皮纸、石材、朱红、深墨等罗马气质，但不得牺牲可读性。

当前确认的主视觉方向为：**v3.25.1 功能模型 + v4.0 深色外壳体系 + 象牙白阶段桌面 + 羊皮纸事务卡片 + 饱满 HUD 控件**。

核心材质层级：

1. **深墨环境层**：应用背景、左侧阶段导航、右侧状态/日志区，负责形成游戏世界的沉浸边界。
2. **深朱红 HUD 层**：顶栏、底部查询栏、关键标题栏，负责全局状态与全局入口。
3. **象牙白阶段桌面层**：中央阶段区的主背景，模拟共和国议事桌面，是玩家当前处理事务的工作空间。
4. **羊皮纸事务卡层**：候选人、法案、战争、收入项、事件与提示等具体对象，以卡片/表格/列表形式摆放在桌面上。
5. **铜金操作层**：当前状态、主按钮、重点数值、边框和 hover 反馈，负责引导玩家行动。

设计判断：不要使用“全羊皮纸”铺满所有区域。全羊皮纸会削弱五区层级，使界面接近历史主题后台系统；深墨外壳与象牙白桌面结合，能更清楚地区分全局环境、当前工作区和具体事务对象。

## 3. 布局规范

推荐采用稳定五区布局：

| 区域 | 职责 | 设计要求 |
| --- | --- | --- |
| 顶栏 | 回合、年份、当前玩家、国库、阶段状态 | 高度固定，信息短句化，不承载复杂操作 |
| 左侧阶段导航 | 7 阶段总流程 | 永远可见，明确当前阶段、已完成阶段、未来阶段 |
| 中央阶段区 | 当前阶段的主要展示与操作 | 最大面积留给阶段业务，不塞全局杂项 |
| 右侧状态栏 | 当前派系、权限、事件日志、流程控制 | 只放辅助判断和流程控制，不承载复杂表格 |
| 底部查询栏 | 全局只读查询入口 | 12 个按钮固定位置，查询结果用浮窗展示 |

顶栏和底部查询栏应采用“饱满 HUD”处理：控件应填满栏位高度，形成稳定游戏 HUD，而不是细小按钮漂浮在空白栏中。顶栏状态块、回合信息、底部查询按钮应具备一致高度、明确边框、半透明深色底和 hover/active 反馈。

禁止事项：

- 不得把所有信息堆到单一大表。
- 不得让阶段导航、查询栏、日志区随着内容变化大幅跳动。
- 不得把查询结果常驻挤占右侧状态栏。
- 不得用大型装饰图压缩阶段操作空间。

## 4. 7 阶段流程规范

GUI 必须保留并清晰展示以下真实阶段顺序：

1. 天命
2. 收入
3. 广场
4. 人口
5. 元老院
6. 战争
7. 决算

阶段导航状态至少包含：

- 当前阶段
- 已完成阶段
- 未解锁阶段
- 可查看但不可操作阶段

阶段页面原则：

- 当前阶段可以操作。
- 非当前阶段通常只读。
- 未实现阶段必须明确显示“后续任务接入”，不得伪装成已完成。
- 每个阶段应有独立“进入下一阶段”或“完成阶段”的流程按钮；执行阶段动作后，不应立即跳走导致玩家看不到结果。

## 5. 信息架构规范

中央阶段区推荐分为三层：

1. 阶段标题层：阶段名、状态、简短说明。
2. 决策信息层：本阶段关键对象、数值、候选项、风险提示。
3. 操作反馈层：执行结果、影响摘要、下一步入口。

对于复杂阶段：

- 元老院：提案、投票、战争、预算、总督任命应拆成标签或分组。
- 广场：招募、竞标、土地交易、舰队建造应分组展示。
- 战争：战争列表、军团、舰队、战斗结果必须分层，不可混在一个区域。
- 收入：收入来源、支出、战争赔款、国库变化应展示为资金流，而不是单行文本。

---

## 6. 查询系统规范

底部 12 个查询按钮应作为全局只读入口存在。

推荐第一批真实接入优先级：

1. 游戏状态
2. 派系信息
3. 战争列表
4. 军团状态
5. 其余按钮先占位或只读简版

查询结果展示方式：

- 使用轻量浮窗。
- 浮窗应包含标题、状态、摘要、列表明细和关闭按钮。
- 查询不得改变游戏状态。
- 查询不得绕过权限边界展示隐藏信息。
- 查询入口不得挤占右侧状态栏常驻空间。

## 7. 视觉规范

### 7.1 色彩

推荐色彩方向：

| 用途 | 建议 |
| --- | --- |
| 应用外壳背景 | 深墨、石板灰、低饱和暖灰 |
| 顶栏/底栏/HUD | 深朱红、深墨、铜金边框 |
| 中央阶段桌面 | 象牙白、低饱和暖白 |
| 事务卡片 | 羊皮纸浅色、重点卡片可略深 |
| 主强调 | 罗马朱红、铜金 |
| 成功 | 稳重绿色 |
| 警告 | 铜橙或赭色 |
| 危险 | 暗红 |
| 禁用 | 低对比灰色 |

注意：

- 不要做单一紫色、蓝色或棕色主题。
- 不要依赖大面积渐变球、光斑、装饰性背景。
- 罗马元素应作为边框、徽记、分隔线、材质暗示，不应喧宾夺主。

### 7.1.1 v3.25.1 视觉基线

后续开发以 `EOR_GUI_Prototype_v3.25.1.html` 的视觉结构为实施基线：

- **保留 v3.25.1 的布局、DOM 结构、功能入口和交互逻辑**，不得为了视觉升级重排五区结构。
- **吸收 v4.0 的美术语言**，但不直接复制 v4.0 的整体 grid 重构。
- 顶栏、左侧阶段导航、右侧状态栏、底部查询栏使用深墨/深朱红外壳。
- 中央阶段区使用象牙白桌面，而不是整块羊皮纸。
- 中央区内的具体事务对象使用羊皮纸卡片，体现“桌面上摆放共和国文书”。
- 卡片标题、主操作按钮、当前阶段提示使用朱红与铜金强调。
- 视觉效果应克制：允许阴影、边框、轻微材质感；避免大面积光斑、过重渐变和纯装饰图案。

推荐色彩参考：

| 层级 | 建议色彩方向 | 用途 |
| --- | --- | --- |
| 深墨环境 | `#15120E` / `#1F1812` / `#241D16` | 应用背景、侧栏、状态区 |
| 深朱红 HUD | `#6B1C00` / `#7B220E` / `#8B2500` | 顶栏、底栏、标题栏、主按钮 |
| 象牙白桌面 | `#F1EDE3` / `#F3EEE4` | 中央阶段主工作区 |
| 羊皮纸事务卡 | `#FBF3DF` / `#FFF7E8` / `#F3E2B8` | 卡片、表格、事务对象 |
| 铜金强调 | `#B99145` / `#D9AF63` / `#F2D590` | 边框、当前状态、hover、重点数值 |

### 7.2 字体

推荐方向：

- 中文正文使用清晰无衬线字体。
- 标题可使用稍具碑刻感或议事厅感的字体，但必须保证可读。
- 阶段标题不应过大，避免挤压操作区。
- 表格、数字、状态值应使用更紧凑、更稳定的字号。

字号建议：

| 类型 | 建议 |
| --- | --- |
| 顶栏状态 | 12-14px |
| 阶段标题 | 22-28px |
| 卡片标题 | 15-18px |
| 正文 | 13-15px |
| 表格/列表 | 12-14px |
| 按钮 | 13-15px |

## 8. 组件规范

### 8.1 阶段导航

阶段导航必须：

- 固定位置。
- 清晰显示编号和名称。
- 当前阶段高亮。
- 已完成阶段有稳定标识。
- 不用大段说明文字。

### 8.2 操作按钮

按钮应分层：

- 主操作：阶段核心动作。
- 次操作：查看、刷新、返回。
- 危险操作：解散、否决、宣战等，需要明确颜色或确认。
- 禁用操作：保留位置但降低透明度，并说明原因。

### 8.3 状态卡片

状态卡片用于展示：

- 当前玩家
- 派系金库
- 国库
- 当前阶段
- 权限提示
- 战争风险
- 事件摘要

卡片应短句化，不写长段说明。

### 8.4 日志

日志用于解释刚刚发生的事情，不替代主要结果展示。

日志规则：

- 最新事件在上方。
- 每条日志包含时间/阶段/事件摘要。
- 关键结果应同时在阶段主区展示。

### 8.5 顶栏与底部查询栏 HUD

顶栏和底部查询栏是全局 HUD，不应表现为普通网页工具条。

顶栏要求：

- 高度固定，但内部控件应尽量填满有效高度。
- Logo、状态块、回合信息应形成三个稳定区段。
- 状态块使用 pill/card 样式，短标签 + 明确数值，不写长句。
- 回合/年份信息应作为右侧醒目胶囊块，不要只是一行裸文本。
- 顶栏不得承载复杂操作。

底部查询栏要求：

- 12 个查询按钮应均分或尽量均分横向空间。
- 按钮高度接近查询栏内部高度，避免小按钮漂浮感。
- 图标与文字居中，hover 时有铜金边框或浅金背景反馈。
- 查询栏只承担全局只读入口，不承载阶段核心动作。
- 在 1366x768 和 1280x720 下，按钮不得挤出窗口或遮挡中央区核心操作。

## 9. 交互规范

### 9.1 阶段动作与阶段推进分离

阶段动作和进入下一阶段必须拆开。

示例：

- 点击“执行天命”后，停留在天命页并展示结果。
- 玩家确认后，再点击“进入收入阶段”。

原因：

- 玩家需要阅读结果。
- GUI 不应让状态跳转显得突然。
- 后续阶段复杂后，这种分离能减少误操作。

### 9.2 当前玩家与权限

GUI 必须随时提示：

- 当前玩家是谁。
- 当前阶段是否由该玩家操作。
- 当前按钮为何不可用。

禁止：

- 靠报错才告诉玩家“不是你的回合”。
- 在 GUI 层绕过 API 权限。
- 直接展示其他玩家隐藏资产或隐藏决策。

### 9.3 浮窗

浮窗适合：

- 查询结果
- 详情展开
- 操作确认
- 错误详情

浮窗不适合：

- 承载完整阶段主流程
- 代替阶段页面
- 常驻占据主布局空间

## 10. i18n 规范

真实 GUI 实现中，所有新增 UI 文案必须集中管理。

要求：

- QML 文案走 `GuiText.get("key")` 或等价集中接口。
- Python/DTO 文案应提供稳定 key，避免 GUI 直接依赖中文文本判断逻辑。
- 不得在 QML 中新增散落中文硬编码。
- 阶段名、按钮名、状态名、错误提示、空状态提示、占位提示均应可翻译。

HTML 原型可以使用静态中文，但转入 QML 实现时必须拆入 i18n 层。

## 11. 架构边界规范

真实 GUI 必须遵守：

```text
GUI/QML
↓
GuiSessionStore
↓
GuiApiAdapter
↓
API
↓
Core/System/Service
↓
Entity
```

禁止路径：

- GUI 直接调用 CLI Command。
- GUI 直接调用 `game_api.execute_phase()` 或 `execute_turn()` 完成复杂阶段。
- GUI 直接读写 Core 私有字段。
- GUI 自行复制元老院、收入、广场、战争等核心规则。
- GUI 为了展示方便绕过多玩家权限。

---

## 12. 可玩性验收标准

一个阶段 GUI 闭环至少满足：

1. 玩家知道当前阶段。
2. 玩家知道自己能否操作。
3. 玩家能看到做决策所需的关键信息。
4. 玩家可以通过 GUI 完成该阶段核心动作。
5. 动作执行后结果可见。
6. 玩家可以明确进入下一阶段。
7. 操作失败时有清楚原因。
8. CLI、自动模式、多玩家权限不退化。

---

## 13. 视觉验收标准

人工验收时重点检查：

- 1366x768 下是否可用。
- 1280x720 下是否不遮挡核心操作。
- 1920x1080 下是否不过度空旷。
- 7 阶段导航是否一眼可懂。
- 底部 12 查询按钮是否清楚、饱满、均分，并且不喧宾夺主。
- 顶栏状态块是否填满栏位、层级清楚，而不是小文本漂浮在顶栏中。
- 中央阶段区是否呈现“象牙白桌面”，具体事务对象是否呈现“羊皮纸卡片”。
- 查询浮窗是否可读、可关闭、不遮挡关键永久状态。
- 文本是否溢出按钮或卡片。
- 当前玩家、当前阶段、下一步入口是否明显。
- 禁用按钮是否说明原因。

## 14. Skill 与工作流考虑

本设计遵循以下工作流思想：

### 14.1 SA 架构审查视角

界面不是单纯美化，而是 API 边界和状态流的可视化。任何 GUI 设计都必须服务于：

- 阶段职责清楚。
- 玩家权限清楚。
- API 调用路径清楚。
- 状态变化可追踪。

### 14.2 PM 任务拆分视角

GUI 不应一次性大重写，应拆成可验收小任务：

1. 主壳与导航。
2. 只读查询。
3. 单阶段闭环。
4. 复杂阶段分批交互化。
5. 一局流程验收。
6. 视觉与 i18n 收束。

### 14.3 前端体验视角

策略游戏 GUI 应优先支持反复操作、扫描比较和结果理解，而不是营销式页面。

因此：

- 少用大 Hero。
- 少用纯装饰卡片。
- 多用稳定区域、清楚列表、明确按钮、结构化反馈。

### 14.4 人工 GUI 验证视角

自动截图可以辅助，但最终 GUI 体验以项目负责人真实窗口手工验证为准。

原因：

- 字体渲染、中文显示、窗口比例、点击反馈很难仅靠自动测试判断。
- 自动测试负责边界和启动，人工验证负责真实体验。

## 15. 后续落地建议

建议将 `EOR_GUI_Prototype_v3.25.1.html` 作为后续 OC 开发的当前实施基线，将 `EOR_GUI_Prototype_Codex_v4.0.html` 作为视觉质量和组件质感参考样张。

后续落地顺序：

1. 以 v3.25.1 保持现有界面布局、功能入口、阶段流程和交互逻辑。
2. 按本规范固化深墨外壳、象牙白阶段桌面、羊皮纸事务卡、饱满 HUD 的视觉体系。
3. SA 将本规范转化为 GUI 后续子任务约束。
4. CGT-01 / OC 按小任务迁移到 QML，不直接复制 HTML，但应保持可见结构与体验一致。
5. 每个阶段完成后进行人工 GUI 验证。
6. 最后做 i18n、视觉一致性和多分辨率收束。

## 16. v3.25.1 更新决策记录

本次规范更新固化以下已确认决策：

1. v3.25.1 是当前 GUI 开发实施基线。
2. v4.0 是视觉方向参考，不直接替代 v3.25.1 的布局和功能模型。
3. 后续视觉升级不得改变 v3.25.1 的界面布局、阶段流程、DOM/组件职责和功能入口。
4. 采用“深墨外壳 + 深朱红 HUD + 象牙白阶段桌面 + 羊皮纸事务卡 + 铜金强调”的材质体系。
5. 不采用“全羊皮纸”界面，因为它会削弱五区层级和游戏 HUD 感。
6. 顶部与底部控件应更饱满，填满顶栏和查询栏，形成稳定游戏 HUD。
7. 具体卡片应使用羊皮纸质感；阶段主区应使用象牙白桌面质感。
8. 所有修改仍必须服从 API 边界、权限边界、阶段动作与阶段推进分离等架构规范。

## 17. 一句话原则

EOR 的 GUI 应该让玩家感觉自己是在主持一个复杂但有秩序的共和国，而不是在点击一组从 CLI 搬来的按钮。
