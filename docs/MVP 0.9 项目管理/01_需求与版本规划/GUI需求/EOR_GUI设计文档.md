## 2. 多玩家交互流程（关键设计）

### 2.1 玩家体系总览

**支持人数：** 单机 3-6 名玩家，各控制一个派系

**玩家类型：**

| 类型 | 操作方式 | UI 表现 |
|------|---------|--------|
| **人类玩家** | 手动操作，轮流进行 | 显示当前玩家标记、可操作控件 |
| **AI 玩家** | 后台自动决策，调用 Auto*Decider | 操作过程不展示，直接显示结果 |

**AI 处理原则：**
- AI 玩家不占用 UI 操作轮次
- AI 操作在人类玩家操作间隙自动执行
- 人类操作完毕后，后台执行所有 AI 玩家的操作，然后推进到下一子环节
- UI 中 AI 操作结果显示为系统消息或日志条目

**示例 — 广场阶段解雇环节：**
```
人类玩家 Optimates 操作 → 点「⏭️ 完成」
         ↓
AI Populares 自动解雇（后台执行，不展示中间过程）
         ↓
AI Equites 自动解雇（后台执行）
         ↓
所有玩家完成 → 自动解锁下一子环节
         ↓
公示区显示解雇结果汇总 + 日志记录 AI 操作
```

### 2.2 各阶段玩家操作角色

| 阶段 | 操作者 | AI 模式 | 说明 |
|------|--------|---------|------|
| 天命 | **任意玩家** 执行一次 | 自动抽取事件 | 谁点都一样，结果全局共享 |
| 收入 | **任意玩家** 确认结算 | 自动结算 | 无需轮流，确认即完成 |
| 广场 | **所有玩家轮流** | 自动执行退役/招募/竞标 | 子环节1→轮流→解锁→子环节2→轮流 |
| 人口 | **所有玩家轮流** | 自动庆典/投票 | 子环节1→轮流→解锁→子环节2→轮流 |
| 元老院 | **执政官**提案 → **所有玩家**表决 → **保民官**否决 | 自动提案/表决/否决 | 三环节有角色锁定 |
| 战斗 | **任意玩家** 依次进攻各战争 | 自动裁定 | 各战争独立，互不影响 |
| 决算 | **自动执行** | 自动 | 无玩家操作 |

### 2.3 子环节状态机

```
[锁定状态] → 所有玩家完成前置子环节 → [解锁状态]
       ↓                                        ↓
  半透明+模糊+锁图标                      可正常操作
  显示"等待XX完成"                       显示操作内容
  （AI 环节时无此界面）
```

**单玩家模式：** `bypass_player_check=True` 时，跳过玩家轮流检验，点一次按钮直接结束子环节。

### 2.4 玩家切换机制

```
当前玩家操作中：
┌─────────────────────────────────────────┐
│ 🎯 当前玩家: Optimates · Marcus Fabius  │
│                                        │
│  [操作内容...]                          │
│                                        │
│  ⏭️ 完成操作 → 移交给下一位             │
│  ──────── 下一位: Populares ────────    │
└─────────────────────────────────────────┘

人类全部完成 → AI 后台执行 → 所有完成 → 解锁子环节
```

**后端调用：**

```python
player_api.next_player(state, player_id)  # 切换到下一位玩家
state.is_current_player(viewer_player_id)   # 判断是否该玩家操作
```

### 2.5 信息可见性规则
| 信息类型 | 可见范围 |
|---|---|
| 国库金额 | **所有玩家**可见 |
| 派系金库 | **仅本派系**可见 |
| 派系总收入 | **所有玩家**可见（收入阶段公示区） |
| 人物私产 | **仅本人/本派系**可见 |
| 招募结果 | 显示被招募人物，**不显示招募者** |
| 合同竞标结果 | 显示中标人物和金额 |
| 土地认购结果 | 显示认购量和退回量 |
| 战斗结果 | **所有玩家**可见 |
| 选举结果 | **所有玩家**可见 |

## 3. 各阶段详细设计

### 3.1 天命阶段

**类型：** 自动执行（无玩家操作）

**多玩家操作：**
- **操作者：** 任意一名玩家（或 AI）点击即可，结果全局共享
- **AI 模式：** 自动执行，无需 UI 操作
- 按钮点击一次后灰化，不可重复点击
- 单次操作全局生效，其他玩家无需额外操作

**UI 要求：**
- 展示一个圆形居中金色按钮「神的旨意」
- 点击执行后按钮灰化，不可再次点击
- 仅触发一个随机事件（不是多个）
- 死亡事件为猝死，需明确显示：
  - 死者名称
  - 所属派系
  - 损失土地数量（收归国库）
  - 损失财富数量（收归国库）
- 事件结果在主界面直接展示（非弹窗）
- 执行后显示「进入收入」按钮，推进到下一阶段

### 3.2 收入阶段

**子环节：1个（查看/确认）**

**多玩家操作：**
- **操作者：** 任意一名玩家（或 AI）点击确认即可
- **AI 模式：** 自动结算，无需 UI 操作
- 各派系财政数据仅本派系可见（派系金库信息）
- 总收入数据所有玩家可见
- 确认一次后全局推进

**UI 要求：**
- 国家收入（左）和国家支出（右）左右并排
- 派系财政列表与地主收入列表**左右并排**（非上下排列）
- 派系财政：仅显示各派系总收入和拨款，不展示人物私产
- 地主收入：可点击人物查看详情，但不显示其财富
- 底部显示国库净变化和新余额
- 点击「确认收入结算」后按钮灰化

### 3.3 广场阶段

**子环节：2个（并排）**

#### 多玩家交互流程

广场阶段是**最典型的多玩家轮流操作**的阶段。CLI 层使用 5 步状态机，GUI 通过并排面板+玩家切换实现同等功能：

```
CLI 5 步:   公告 → 裁员(轮流) → 市场(轮流) → 公示 → 私地交易
               ↓       ↓             ↓          ↓        ↓
GUI 2 面板: [公示区]    [子环节1: 解雇]   [子环节2: 人才市场]   [公示区]   ⏭️ 续
```

**玩家切换机制：**

每个子环节内部，玩家按固定顺序轮流操作：

```
子环节1（解雇）：
  ┌────────────────────────────────────────┐
  │ 🎯 当前玩家: Optimates · Marcus Fabius │
  │                                        │
  │ [✓] Gaius Manlius (Equ · 影响力3)  解雇 │
  │ [ ] Numerius Antonius (Equ · 影响力2)   │
  │                                        │
  │    ⏭️ 完成解雇（当前玩家）              │
  │    ─── 下一位: Populares · Lucius Julius ───
  └────────────────────────────────────────┘
```

**流程细节：**

| 步骤 | 谁操作 | 操作内容 | 切换方式 |
|------|--------|---------|---------|
| 1. 子环节1（解雇） | 玩家 1 → 玩家 2 ... 逐个轮流 | 勾选本派系成员 → 解雇 | 各玩家点击「⏭️ 完成解雇」→ 移交给下一位 |
| 2. 子环节1 结束 | 所有玩家完成 | 自动将解雇人物加入人才市场 | 自动解锁子环节2 |
| 3. 子环节2（市场） | 玩家 1 → 玩家 2 ... 逐个轮流 | 招募/竞标/认购/凯旋 | 各玩家点击「⏭️ 完成市场」→ 移交给下一位 |
| 4. 子环节2 结束 | 所有玩家完成 | 公示区自动更新竞价结果 | 自动完成广场阶段 |

**后端 API 支持：**

```python
player_api.next_player(state, player_id)  # 切换到下一位玩家
state.is_current_player(viewer_player_id)   # 判断当前是否是该玩家的回合
```

**子环节1 各玩家可见内容：**
- 仅显示**本派系**成员列表
- 仅可解雇本派系成员

**子环节2 各玩家可见内容：**
- 人才市场（所有派系可见）
- 合同信息（所有派系可见）
- 待售公地配额（所有派系可见）
- 凯旋投票（所有派系可见）
- 各派系金库（仅本派系可见）

#### 公示区内容
- 阶段公告信息（当前战争/合同/凯旋概况）
- 所有玩家操作完毕后自动更新竞价结果
- 招募结果：显示**被招募的人物**（不显示招募者身份）
- 合同竞标结果：显示中标人物和金额
- 土地认购结果：显示认购量和退回公地池的量
- 凯旋投票结果（凯旋仪式在人口阶段执行，此处仅显示投票通过/否决）

#### 子环节1：解雇成员
- 显示本派系成员列表（姓名、派系、影响力）
- 勾选要解雇的成员
- 点击「解雇」→ 从解雇列表中移除 → **自动刷新到人才市场**
- 按钮：「解雇」+「关闭」，左右并排
- 当前玩家点击「⏭️ 完成解雇」→ 移交下一位玩家
- 所有玩家完成后 → 子环节2自动解锁

#### 子环节2：人才市场（招募/竞标/认购/否决/凯旋）

**人才市场表格（7列）：**
```
姓名 | 军略 | 智略 | 魅力 | 热忱 | 阶级 | 费用
```
- **表格形式和内容保持 V3 原样**，仅重布局
- 阶级映射：Equites→骑士，Optimates→贵族，Populares→平民
- 属性用纯数字（无图标）
- 招募按钮点击后灰化

**合同竞标：**
- 显示待竞标合同（ID、名称、预算）
- 点击「竞标」弹出弹窗输入金额
- 弹窗按钮：「确认」+「关闭」左右并排

**公地认购：**
- 操作形式保持 V3 原样
- 弹出弹窗输入认购数量（上限由可用公地决定）
- 弹窗按钮：「确认」+「关闭」左右并排

**凯旋投票：**
- 赞成/否决互斥，只能选一次
- 点击后两个按钮同时灰化，选中者高亮

**公示结果自动更新（子环节2完成后）：**
- 招募结果：显示**被招募的人物**（不显示招募者身份）
- 合同竞标结果：显示中标人物和金额
- 土地认购结果：显示认购量和退回公地池的量
- 凯旋投票结果（凯旋仪式在人口阶段执行，此处仅显示投票通过/否决）

**⏭️ 完成本次操作：** 点击后轮到下一位玩家；所有玩家操作完毕后公示区自动更新

### 3.4 人口阶段

**子环节：2个（并排）**

**多玩家操作：**

```
子环节1（庆典赞助）→ 按玩家轮流 → 全部完成 → 解锁子环节2
子环节2（投票选举）→ 按玩家轮流 → 全部完成 → 结算选举结果
```

| 步骤 | 谁操作 | 操作内容 | 切换方式 |
|------|--------|---------|---------|
| 子环节1 庆典 | 玩家 1 → 2 → ... 逐个轮流 | 为本派系候选人输入赞助金额 | 各玩家「⏭️ 完成庆典」→ 下一位 |
| 交接 | 全部人类完成 | AI 自动为各自派系庆典（后台） | 自动解锁子环节2 |
| 子环节2 投票 | 玩家 1 → 2 → ... 逐个轮流 | 选各官职候选人 | 各玩家「⏭️ 完成投票」→ 下一位 |
| 结算 | 全部人类完成 | AI 自动投票（后台）→ 执行选举运算 | 公示区更新结果 |

**AI 模式：**
- AI 调用 `AutoFestivalDecider` 自动决定庆典赞助
- AI 调用 `AutoVoteDecider` 自动投票
- 人类完成子环节后，AI 在后台自动执行

**各玩家可见内容：**
- 候选人列表（所有玩家可见）
- 本派系候选人个人信息（仅本派系可见）
- 庆典赞助金额输入（仅本派系候选人）
- 投票选择（所有派系候选人可见）

#### 公示区
- 候选人表格（9列 flex 对齐）：

```
官职 | 候选人 | 军略 | 智略 | 魅力 | 热忱 | 影响力 | 派系 | 选举结果
```

- 选举前显示候选人信息
- 选举后更新「选举结果」列（当选/落选）+ 公示区顶部显示派系影响力变化
- 每个候选人在独立行，多候选人办公室逐行排列

#### 子环节1：庆典赞助

**规则：**
- 每个派系**只能赞助本派系成员**
- 赞助费来自候选人**个人财富**（非派系金库）
- 每人可赞助多个本派系候选人

**输入格式（表格）：**
```
候选人 | 竞选官职 | 个人财富 | 赞助金额
```
- 赞助金额输入框有上限（max = 个人财富）
- 无「确认」「关闭」按钮
- 当前玩家输入完成后点击「⏭️ 完成庆典」→ 移交下一位玩家
- AI 玩家自动处理，无 UI 操作

#### 子环节2：投票选举

**规则：**
- 玩家可以为**任意派系**候选人投票
- 每个官职单选一名候选人
- **不选 = 弃权**（无显式弃权选项）
- 投票时不显示候选人属性（属性已在公示区表格展示）

**操作：**
- 按官职列出所有候选人 + 派系
- 无「确认」「关闭」按钮
- 当前玩家点击「⏭️ 完成投票」→ 移交下一位玩家
- 所有玩家完成后 → 执行选举计算 + 更新公示区表格中的选举结果 + 更新影响力
- AI 玩家自动投票，无 UI 操作

### 3.5 元老院阶段

**子环节：3个（从左到右并排）**

#### 界面布局
```
┌────────────────────────────────────────────────────────────────┐
│  公示区（顶部通栏）                                            │
│  🏛️ 元老院议事                                               │
│  会议主持 / 席位占比                                           │
│  仅展示：最终通过法案 + 政府运作结果                           │
├───────────────┬──────────────────┬──────────────────────────┤
│  ① 执政官提案   │  ② 元老院表决      │  ③ 保民官否决             │
│               │                  │                          │
│  【初始：可操作】 │  【初始：锁定】    │  【初始：锁定】           │
│  提交后 → 灰化  │  表决后 → 灰化     │  否决后 → 灰化            │
│  解锁环节2     │  解锁环节3/逃生    │  公示最终结果             │
└───────────────┴──────────────────┴──────────────────────────┘
```

#### 子环节1：执政官提案（带参数配置）

- **操作者：** 获得执政官的派系玩家
- **交互方式：** 展开式配置，每个法案点击展开/收起
- **锁定/解锁：** 初始可操作，提交后灰化并解锁子环节2

**8种法案类型及参数配置：**

| # | 类型 | 参数配置 | UI 控件 | 是否需表决 |
|---|---|---|---|---|
| 1 | ⚔️ **宣战** | 征召军团数量（2-10个，显示国库可支撑量） | 下拉选择 | ✅ 需元老表决+保民官否决 |
| 2 | ☮️ **停战** | 显示条约预设条款（赔款/停战期/领土割让） | 只读展示 | ✅ 需元老表决+保民官否决 |

- **提交：** 点击「🗳️ 确认表决 → 移交否决环节」

**边界案例处理（防死锁）：**

```

  └── 是（全部被否）

       ├── 显示 ⚠️ 所有法案均被否决
       ├── 显示 ⏭️ 结束元老院阶段 逃生按钮
       ├── 子环节3标记为 ❌ 无法案通过无需否决
       └── 流程提前终止
```

#### 子环节3：保民官否决

- **法案列表来源：** 自动同步子环节2中同意的法案
- **操作：** 每个法案勾选否决（可多选），未勾选=同意
- **提交：** 点击「🛡️ 确认否决 → 公示结果」

**边界案例处理（防死锁）：**

```
保民官否决 → 全部被否决（存活数 = 0）？
  ├── 否（至少1项存活）→ 公示区显示完整政府运作结果
  └── 是（全部否决）
       ├── 否决面板内追加 ⏭️ 结束元老院阶段 逃生按钮
       └── 公示区显示 ⚠️ 所有法案均被否决
```

#### 公示区（顶部通栏）

- **展示原则：** 只展示**最终结果**，中间过程不在公示区显示
- **最终结果内容：**

**多玩家操作：**
- **操作者：** 任意一名玩家点击某一场战争的「发动进攻」即可
- 各战争独立，多名玩家可各自操作不同战争
- 同一战争每回合仅一次进攻机会，无论谁点击
- AI 玩家：调用 `AutoWarDecider` 自动决定是否进攻
- 所有战争操作完毕后（或无人操作），推进到下一阶段

**操作规则：**
- **进攻次数：** 每场战争每回合**仅一次**进攻机会
- **按钮状态：** 点击后灰化（`disabled` + 降低透明度）
- **结果展示：** 按钮下方区域展示完整战斗裁定，包括：
  - 🎲 掷骰值 + 军力计算（罗马战力 + 骰子 vs 敌方战力）
  - 🏆/⚔️/🤝/😤 战斗结果文字
  - 📜 停战草案（仅平局时：赔款/停战期/领土变更信息）

**战斗裁定（4种结果）：**

```
战斗总战力 = 掷骰(1-6) + 罗马战力
偏差值 = 总战力 - 敌方战力

偏差 ≥ 8  → 🏆 大胜    → 战利品入库
偏差 ≥ 2  → ⚔️ 险胜    → 守住阵地
偏差 ≥ -2 → 🤝 平局    → 生成停战草案（带详细条款）→下回合元老院表决
偏差 < -2 → 😤 败退    → 稳定度 -5%
```

**边界案例：**
- 若第4场战争激活 → 游戏结束（面板占位提示「第4场=游戏结束」）
- 最多3个战争面板，第3个面板为空位占位

## 4. 通用 UI 规范

### 4.1 表格
- 所有表格使用 **flex 对齐**方式（非 HTML `<table>`）
- 表头和数据行使用相同 flex 比例，确保严格对齐
- 参考：`talent-header`/`talent-row`/`cand-header`/`cand-row`/`fest-header`/`fest-row` CSS 类

### 4.2 按钮
- 所有可操作按钮**仅一次有效**，点击后灰化（`disabled` + 降低透明度）
- 按钮文字统一：「确认」+「关闭」（左右并排，不上下排列）
- 凯旋投票：赞成/否决互斥

### 4.3 弹窗
- 弹窗底部按钮行：`「确认」+「关闭」` 水平排列
- 使用 `.modal-btn-row` CSS 类（`display:flex; gap:6px; justify-content:flex-end`）
- 弹窗模版移除底部独立的「关闭」按钮，统一在内容底部行内

### 4.4 属性表示
- **不使用图标/emoji** 表示人物属性
- 属性全部使用**纯文字**：
  - 表头：军略 / 智略 / 魅力 / 热忱
  - 数据：纯数字
- 多候选人属性使用 `<br>` 换行，每行对应一位候选人

### 4.5 子环节锁定
- 子环节2/3 初始为**锁定状态**，需前置子环节全部完成后才解锁
- 锁定效果：半透明（opacity: 0.5）+ 模糊（filter: blur） + 中央提示文字
- 解锁由系统自动完成（所有玩家完成前置环节后→自动解锁）

### 4.6 多语种支持（i18n）

**基本原则：** 所有面向玩家的界面文字均不硬编码，通过 i18n 键值系统加载。

#### 语言支持范围
- 当前支持：简体中文（zh-CN）、英文（en）
- 后端体系：`data/i18n/{language}.json` JSON 文件 + `src/core/i18n.py` 单例加载
- GUI 层：`src/ui/gui/localization.py` 提供适配

#### 设计要求

**所有 UI 文字必须使用 i18n key：**

```py
# ❌ 错误：硬编码
button.text = "发动进攻"
label.text = "国库: 142 T"

# ✅ 正确：通过 i18n 键值
button.text = i18n.get("ui.combat.attack_button")
label.text = i18n.get("ui.treasury.display", amount=142)
```

**需要 i18n 化的文字类型：**

| 类别 | 示例 | i18n key 命名建议 |
|------|------|------------------|
| 阶段标题 | 天命阶段 / 元老院阶段 | `phase.mortality.title` / `phase.senate.title` |
| 按钮文字 | 发动进攻 / 确认表决 / 提交提案 | `ui.btn.attack` / `ui.btn.confirm_vote` / `ui.btn.submit_bills` |
| 状态提示 | 等待执政官提交法案 / 本回合已进攻过 | `ui.status.waiting_consul` / `ui.status.already_attacked` |
| 标签/表头 | 军略 / 智略 / 魅力 / 热忱 | `ui.label.martial` / `ui.label.intellect` / `ui.label.charisma` / `ui.label.zeal` |
| 属性名 | 国库 / 派系 / 活跃战争 | `ui.resource.treasury` / `ui.resource.faction` / `ui.resource.active_wars` |
| 战斗结果 | 大胜 / 险胜 / 平局 / 败退 | `ui.combat.result.triumph` / `ui.combat.result.victory` / `ui.combat.result.stalemate` / `ui.combat.result.defeat` |
| 边角提示 | ※ 点击各战争面板... / ⏳ 等待子环节1完成 | `ui.hint.click_to_attack` / `ui.status.locked_waiting` |
| 弹窗文字 | 详情 / 确认 / 关闭 | `ui.modal.title.detail` / `ui.btn.confirm` / `ui.btn.close` |

**带参数的动态文字：**

```py
# 字符串模板中的 {0}、{amount} 等占位符
"国库: {amount} T"  →  i18n.get("ui.treasury.display", amount=142)
"派遣 {0} 个军团" →  i18n.get("ui.senate.legions_count", count=6)
"{name} 加入派系"  →  i18n.get("ui.forum.recruited", name="Gaius")
```

**HTML 原型中的处理：**
- 原型中的硬编码文字仅为设计演示
- 实际代码实现时，所有字符串替换为 `i18n.get()` 调用
- 原型中的模拟数据（人物名、数字、事件）仍可保留硬编码，仅标注为 mock
