# AI 开发任务模板

适用对象：DeepSeek、Codex 或其他承担具体模块开发的 AI。

适用阶段：MVP 0.7 架构收口、MVP 0.9 / MVP 1.0 P1 功能开发。

P1 功能任务必须同时参考：

`E:\Eagle of Rome\MVP 0.9 项目管理\P1功能最小可用GUI开发与验收标准.md`

## 任务基本信息

| 字段 | 内容 |
| --- | --- |
| 任务编号 | 例如 `TD-2026-03-30-001` / `P1-04.0` |
| 任务名称 | 例如 `PoliticalSystem 重构` |
| 任务类型 | 架构重构 / 功能开发 / 缺陷修复 / 测试补齐 / 文档同步 |
| 优先级 | P0 / P1 / P2 |
| 负责人 AI | DeepSeek / Codex / 其他 |
| 预计工时 | 例如 0.5 天 / 2 天 / 3 天 |
| 当前基线 | 例如 `mvp-0.7.44-final-20260330` |
| 目标版本 | 例如 MVP 0.9 / MVP 1.0 |

## 背景说明

说明为什么要做这个任务：

- 当前问题是什么。
- 对玩家体验、架构稳定性或后续功能有什么影响。
- 是否来自技术债务、测试失败、文档差异或新功能规划。

## 允许修改范围

明确列出允许修改的文件或目录。

示例：

- `src/core/systems/political_system.py`
- `src/api/senate_api.py`
- `src/ui/commands/phase_senate.py`
- `src/tests/test_systems/test_political_system.py`
- 相关文档或函数索引

## 禁止事项

除非任务明确授权，否则禁止：

- 直接修改实体或 `GameState` 的私有字段。
- 在 UI 命令层新增核心业务规则。
- 让实体层依赖 API 层、UI 层或命令层。
- 绕过 `forum_api`、`senate_api`、`population_api` 等标准 API。
- 新增与既有模式不一致的返回格式。
- 删除旧逻辑而不提供兼容路径或测试证明。
- 修改无关文件或做大范围格式化。

## 实现要求

说明具体要完成什么。

示例：

1. 新增或调整核心系统/服务类。
2. 将原 UI/API 中分散的业务逻辑迁入系统/服务层。
3. API 层只做权限校验、参数校验、调用系统/服务、返回统一格式。
4. UI 命令层只做输入解析、调用 API、格式化输出。
5. 保持自动模式和手动模式行为一致。
6. 保持多玩家信息隔离逻辑不退化。

## 调试日志要求

新增或改动关键流程时必须使用 `GameState.log_event()` 记录。

至少覆盖：

- 函数入口和出口。
- 关键状态变化。
- 决策器输入、随机数和决策结果。
- API 调用结果。
- 一致性校验失败。
- 资金、合同、战争、舰队、玩家切换等关键路径。

DEBUG 日志建议包含结构化 `extra` 字段：

```python
state.log_event(
    "[DEBUG] 说明文字",
    level=logging.DEBUG,
    extra={
        "function": "function_name",
        "phase": "enter",
        "object_id": object_id,
        "success": True,
    },
)
```

## 测试要求

必须说明测试范围。

| 测试类型 | 要求 |
| --- | --- |
| 单元测试 | 覆盖正常路径、异常路径、边界条件 |
| 集成测试 | 覆盖阶段流程或跨模块调用 |
| 回归测试 | 确保 MVP 0.7 既有流程不破坏 |
| 手动冒烟 | 如涉及 CLI 交互，说明手动验证步骤 |

运行测试后必须给出：

- 测试命令。
- 通过/失败数量。
- 失败原因和是否已修复。

## P1 功能最小可用 GUI 要求

涉及 MVP 0.9 / MVP 1.0 P1 功能时，任务书必须填写：

| 字段 | 内容 |
| --- | --- |
| GUI 交付分级 | A 交互型 / B 被动状态型 / C 底层支撑型 / D 开发工具型 |
| 玩家入口 | 从哪个 GUI 页面、阶段或流程进入 |
| 决策信息 | 玩家完成操作前必须看到哪些信息 |
| 核心操作闭环 | 玩家无需 CLI 可以完成的操作 |
| 结果反馈 | 成功、失败、取消和状态变化如何呈现 |
| 状态刷新 | 哪些数值、列表和阶段状态必须同步更新 |
| 权限边界 | 当前玩家可见和可操作范围 |
| API 边界 | GUI 调用的 API 及返回结构 |
| 兼容要求 | CLI 和自动模式需要保留的行为 |
| GUI 测试 | 成功、非法操作、状态刷新、权限隔离测试 |
| 非本轮内容 | 动画、完整迁移等明确延期内容 |

若不交付 GUI，必须说明属于 C 或 D 类，并指定未来承接界面的功能或任务。玩家仍需切回 CLI 才能完成 A 类功能时，不得验收通过。

## 验收标准

任务完成必须满足：

- 功能行为符合依据文档。
- 分层依赖符合 `UI -> API -> Core/System/Service -> Entity`。
- 无新增私有字段直改。
- 资金、合同、战争、舰队等状态保持一致。
- 自动模式与手动模式均可运行。
- P1 玩家功能达到《P1功能最小可用GUI开发与验收标准》；玩家无需切回 CLI 完成核心决策闭环。
- 相关测试通过。
- 相关函数索引或文档已同步。

## 交付物

AI 输出必须包含：

- 修改文件清单。
- 核心实现摘要。
- 测试结果。
- 已知风险。
- 未完成事项。
- 是否需要项目经理复核。

## 推荐输出格式

```text
一、完成内容
二、修改文件
三、测试结果
四、架构合规说明
五、风险与遗留问题
六、建议下一步
```
