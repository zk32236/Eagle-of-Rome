## 2. 架构合规检查

| 检查项 | 通过标准 | 结果 |
| --- | --- | --- |
| 分层方向 | 依赖方向保持 `UI -> API -> Core/System/Service -> Entity` |  |
| UI 命令层职责 | 只解析输入、调用 API、格式化输出 |  |
| API 层职责 | 统一权限校验、参数校验、调用核心逻辑、返回标准格式 |  |
| 系统/服务层职责 | 承载政治、经济、战争、舰队等核心业务规则 |  |
| 实体层职责 | 只维护自身状态和局部行为，不依赖 UI/API |  |
| 决策器职责 | 外部输入和 AI 决策通过 decider 接口隔离 |  |
| 多玩家隔离 | 玩家切换仍通过 `Command` 基类方法处理 |  |

## 3. 禁止事项检查

以下任一项出现，原则上不得验收通过，除非任务书明确授权并说明原因。

| 禁止项 | 是否出现 | 位置 |
| --- | --- | --- |
| UI 层直接修改 `GameState` 私有字段 |  |  |
| UI 层直接修改实体私有字段 |  |  |
| API 层重复定义统一工具函数 |  |  |
| API 层反向导入 UI 命令类 |  |  |
| 实体层依赖命令层或 API 层 |  |  |
| 绕过标准 API 直接写临时数据 |  |  |
| 新增旧式返回值而非 `{success, message, data, errors}` |  |  |
| 删除旧逻辑但未补测试 |  |  |
| 修改无关文件或大面积格式化 |  |  |

## 4. 重点技术债务检查

| 债务项 | 验收关注点 | 结果 |
| --- | --- | --- |
| `PoliticalSystem` 重构 | 元老院核心逻辑是否从 UI/API 收束到系统层 |  |
| `EconomicService` 重构 | 收入阶段经济结算是否从命令层迁移到服务层 |  |
| 广场阶段土地交易 API 化 | 是否不再直接写 `_forum_pending` |  |
| `api_response` 统一 | 是否统一从 `src.api` 导入 |  |
| 旧竞标接口 | `GameState.place_bid/resolve_auction` 是否清理、兼容或标记废弃 |  |
| 影子 `Province` | `entities.py` 中旧 `Province` 是否明确废弃或移除 |  |
| 私有字段访问 | 是否新增公共方法替代 `_senate_pending`、`_population_pending` 等直改 |  |

## 5. 功能行为检查

| 模块 | 检查点 | 结果 |
| --- | --- | --- |
| 广场阶段 | 招募、退休、竞标、公地认购、凯旋投票、土地交易流程可运行 |  |
| 人口阶段 | 庆典、投票、`campaign all`、`vote all` 可运行 |  |
| 元老院阶段 | 提案、投票、否决、宣战、停战、总督任命、预算通过可运行 |  |
| 收入阶段 | 私地收益、公地收益、合同收益、军费、海军维护费、赔款可结算 |  |
| 战斗阶段 | 陆战、海战、停战草案、指挥官回归逻辑不退化 |  |
| 决算阶段 | 总督交接、合同到期、军团恢复、临时影响力衰减可运行 |  |
| 多玩家 | 手动多玩家模式不会泄露其他派系信息 |  |
| 自动模式 | AI 决策器能完整跑通对应阶段 |  |

## 6. 数据一致性检查

| 对象 | 检查点 | 结果 |
| --- | --- | --- |
| 资金 | 扣款和入账原子性，失败时不留下半状态 |  |
| 合同 | `PENDING -> BUDGETED -> ACTIVE -> COMPLETED/EXPIRED` 状态流转正确 |  |
| 舰队 | 建造合同完成后舰队可用，合同同步完成 |  |
| 战争 | 威胁、活跃、停战、解决状态列表一致 |  |
| 总督 | 候任、现任、旧任返回逻辑一致 |  |
| 人物 | 死亡、缺席、官职、影响力、财富、私地状态一致 |  |
| 玩家 | 当前玩家、回合顺序、权限校验一致 |  |
