# EOR P1 功能最小可用 GUI 开发与验收标准

版本：V1.0

生效日期：2026-06-25

适用阶段：MVP 0.9 / MVP 1.0 P1 功能开发

适用对象：项目经理 PM、系统架构师 SA、CGT-01 及其他模块开发 Agent

## 一、目的

本标准用于界定 P1 功能开发中的“最小可用 GUI”，避免出现以下情况：

- P1 功能只在 Core、API 或 CLI 中完成，玩家仍必须切回 CLI 才能操作。
- 每个 P1 功能都承担完整 GUI 重构，导致范围失控。
- 所有 P1 功能完成后才首次接入 GUI，集中暴露 API 与界面不匹配问题。

项目采用“增量 GUI 接入 + 最终完整迁移”策略：

1. GUI-P0-01 建立 MVP 0.7 可玩 GUI 骨架和通用交互能力。
2. 此后每个需要玩家查看或操作的 P1 功能，随功能开发交付最小可用 GUI。
3. GUI-P1-02 在主要 P1 功能稳定后执行完整 CLI 到 GUI 迁移与产品化收尾。
4. CLI 继续作为调试、测试和兼容入口，不因 GUI 接入而立即删除。

## 二、核心定义

P1 功能的最小可用 GUI，是指：

> 玩家无需打开 CLI，即可在 GUI 中获得完成该功能所需的信息，独立完成一次合法的核心操作或决策，并看到可靠结果后继续游戏主流程。

只有数据展示、占位页面或跳转 CLI 的按钮，不构成最小可用 GUI。

对于纯被动或只读功能，若不存在玩家主动决策，最小闭环应改为：

> 玩家能够在正确时机看见该功能产生的必要状态、变化原因和结果，并能继续游戏主流程。

## 三、强制验收维度

| 维度 | 最低要求 |
| --- | --- |
| 功能入口 | 玩家能从现有 GUI 导航或阶段流程进入，不依赖隐藏调试入口 |
| 决策信息 | 显示完成决策所需的关键对象、数值、条件、成本和限制 |
| 核心操作 | 覆盖该功能面向玩家的核心操作，不得要求切回 CLI 补完 |
| 确认与取消 | 高影响或消耗资源的操作应支持确认；可撤销的选择应能取消或返回 |
| 合法性校验 | 非法操作不可无提示提交；错误信息应足以让玩家修正选择 |
| 结果反馈 | 明确显示成功、失败及关键状态变化 |
| 状态刷新 | 操作完成后，相关数值、列表、按钮状态和阶段状态同步刷新 |
| 流程衔接 | 完成或放弃操作后可以返回并继续当前阶段或回合 |
| 权限隔离 | 只显示当前玩家有权查看的信息，只允许其执行有权执行的操作 |
| 自动模式 | 自动玩家仍走相同 API 或等价受控入口，行为规则不分叉 |
| 架构边界 | GUI 通过 API 操作，不直接修改 Core、System、Service 或 Entity 私有状态 |
| API 格式 | 新增或调整的 API 保持 `{success, message, data, errors}` 结构 |
| CLI 兼容 | 原 CLI 调试或操作路径不因 GUI 接入而无故退化 |
| 测试 | 覆盖至少一个成功闭环、一个失败或非法操作、状态刷新和权限边界 |

以上维度与具体功能无关，SA 在 P1 技术任务书中不得无说明删除。确实不适用的项目，应在任务书中标记“不适用”并说明理由。

## 四、P1 任务的 GUI 交付分级

| 分级 | 适用情况 | GUI 最低交付 |
| --- | --- | --- |
| A：交互型功能 | 玩家需要选择对象、支付资源、投票、确认或执行操作 | 完整决策闭环，满足全部适用的强制验收维度 |
| B：被动状态型功能 | 功能自动结算，但结果影响玩家决策 | 在正确界面展示状态、变化原因、结果和必要提示 |
| C：底层支撑型功能 | 仅为其他功能提供内部数据或规则，当前无玩家入口 | 可暂不新增独立界面，但须提供 GUI 可消费的 API/DTO，并在依赖功能中完成界面闭环 |
| D：开发工具型任务 | 测试、文档、调试、内部重构 | 默认不要求玩家 GUI；不得借此破坏现有 GUI |

PM 在任务意图包中确定初步分级，SA 在技术边界审查中确认最终分级。

## 五、不属于最小可用 GUI 的内容

除非任务书明确要求，单个 P1 功能的最小 GUI 不强制包含：

- 最终视觉主题、精细动画和特效。
- 完整地图表现或复杂场景演出。
- 高级筛选、排序、搜索和批量操作。
- 拖拽、完整快捷键和高级效率工具。
- 全部历史记录、统计图表或百科说明。
- 对整个相关阶段进行一次性界面重做。
- 删除 CLI 或完成所有旧 CLI 流程迁移。
- 与该 P1 功能无关的响应式、国际化或视觉重构。

这些内容可由 GUI-P1-02 或独立 GUI 优化任务统一处理。

## 六、P1 功能任务书要求

自本标准生效后，每份 P1 功能任务意图包和技术任务书必须增加“最小可用 GUI”章节，至少包含：

1. GUI 交付分级。
2. 玩家使用场景与入口。
3. 决策所需信息。
4. 核心操作闭环。
5. 成功、失败和取消反馈。
6. 状态刷新范围。
7. API 与权限边界。
8. CLI 和自动模式兼容要求。
9. GUI 测试范围。
10. 明确列出的非本轮内容。

若本轮不交付 GUI，任务书必须说明其属于 C 或 D 类，并指出未来由哪个依赖功能或 GUI 任务完成玩家界面。

## 七、标准验收场景

对于交互型 P1 功能，SA 至少应要求并验证：

1. 当前玩家从 GUI 进入功能。
2. GUI 展示合法决策所需信息。
3. 玩家完成一次合法操作。
4. GUI 通过 API 提交，成功后刷新相关状态。
5. 玩家尝试一次非法或权限不足的操作，GUI 给出明确反馈且状态不被污染。
6. 玩家可以取消、返回或继续当前阶段。
7. CLI 与自动模式相关回归测试通过。

涉及多玩家信息时，还必须验证非当前玩家或非本派系信息没有泄漏。

## 八、验收结论规则

| 情况 | SA 建议结论 |
| --- | --- |
| 核心功能和最小 GUI 闭环均通过 | `PASS` |
| 核心功能通过，仅有不影响操作的轻微展示或文档条件项 | `CONDITIONAL_PASS` |
| 玩家仍需切回 CLI 才能完成核心操作 | `RETURN_FOR_REWORK` |
| GUI 绕过 API 或直接修改私有状态 | `RETURN_FOR_REWORK` |
| 权限隔离、状态刷新或失败回滚存在问题 | `RETURN_FOR_REWORK` |
| 本应为 A/B 类功能却未交付 GUI，且无 PM 批准的延期理由 | `RETURN_FOR_REWORK` |

## 九、GUI-P1-02 的边界

GUI-P1-02 不负责为所有 P1 功能首次补做 GUI。它是在主要 P1 功能已经具备最小可用 GUI 后执行的完整迁移与产品化收尾，重点包括：

- 补齐尚未迁移的 MVP 0.7 CLI 玩家流程。
- 统一导航、布局、组件、反馈、加载和异常状态。
- 统一视觉规范、可访问性和必要的响应式适配。
- 补齐跨阶段端到端 GUI 流程。
- 清理重复或临时界面实现。
- 明确 CLI 的长期保留范围和定位。

## 十、任务分工

| 角色 | 职责 |
| --- | --- |
| PM | 确定 P1 功能优先级、初步 GUI 分级和玩家验收目标 |
| SA | 审核 GUI/API/Core 边界，定稿技术任务书和 GUI 验收场景 |
| CGT-01 | 同步实现功能、API、最小 GUI 和测试，不自行扩大完整迁移范围 |
| 项目负责人 | 对玩家可用性进行手工复测，批准重大范围调整和阶段切换 |
