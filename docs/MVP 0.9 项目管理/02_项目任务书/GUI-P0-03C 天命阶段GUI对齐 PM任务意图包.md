## 一、背景说明

GUI 主壳骨架（GUI-P0-03A）和 GuiApiAdapter 基础查询能力（GUI-P0-03B）已完成。当前产品目录中，各阶段 QML 页面是在主壳落地时批量生成的，其中 `MortalityStage.qml`（天命阶段）的布局与《EOR GUI设计文档 V2.0》§3.1 和 V3.23 原型存在明显偏差：

1. **中央操作按钮为 120×120 大圆按钮**，设计规范要求为矩形按钮（深红底+金色文字，非圆形）。
2. **右侧 ContextPanel 缺少「进入收入」推进按钮**，推进操作分散在页面底部。
3. **死亡事件详情展示不够清晰**，设计规范要求明确展示死者名称、所属派系、损失土地数量（收归国库）、损失财富数量（收归国库）。
4. **事件结果模型使用数组而非单事件**，与设计规范「仅触发一个随机事件」不一致。
5. **全局阶段容器边距、信息密度** 需参照前期 Senate 阶段对齐经验继续收窄。

## 二、任务目标

将 `MortalityStage.qml` 和相关 `ContextPanel.qml` 布局与 V3.23 原型及《EOR GUI设计文档 V2.0》对齐。**不修改天命事件逻辑、Core 规则或阶段状态机**，纯 GUI 布局与交互调整。

## 四、PM 初步范围建议

### 4.1 本轮调整内容

| 模块 | 当前状态 | 目标 |
|------|---------|------|
| 中央操作按钮 | 120×120 大圆按钮（radius=60），金色边框，⚡ + 文字上下排列 | 改为矩形按钮，深红底色+金色文字，宽高约 180×52，居中独立展示 |
| 按钮回馈 | 执行后按钮灰化，文字改为「✅ 已执行」 | 执行后按钮灰化+红勾，文字改为「已完成」；底部显示绿色「⏭️ 进入收入」推进按钮 |
| 「进入收入」推进按钮 | 当前在底部 RowLayout 中与 execBtn 共享位置区域 | 移至 ContextPanel 右侧栏，作为阶段推进的标准入口；底部保留作为辅助入口 |
| 事件结果展示 | 使用 `sessionStore.mortalityEvents` 数组 + Repeater 渲染多事件列表 | 改为单事件结果视图（设计规范要求仅触发一个随机事件）；若 API 仍返回数组，取第一项展示 |
| 死亡事件详情 | 通过 `modelData` 泛化字段渲染 | 明确展示：死者名称、所属派系、损失土地数量（收归国库）、损失财富数量（收归国库） |
| 阶段容器边距 | 当前 ColumnLayout margin=20, spacing=12 | 参照 Senate 阶段对齐经验压缩至 margin=8~12, spacing=4~8 |
| 步骤指示条 | 逻辑一致，UI 样式可微调 | 保持现有逻辑，视觉对齐原型 |
| 阶段徽标 | 逻辑一致 | 保持，无需修改 |

### 4.2 本轮明确不做

- 不改天命事件类型、触发概率、Core 规则
- 不改 `GameApi.execute_phase()` 或阶段状态机

- 不改 AI 模式（自动执行逻辑）
- 不改 i18n 字符串体系，新增文案走 `GuiText` 集中管理

## 七、验收标准

1. 中央天命执行按钮改为矩形（深红底+金色文字），非圆形
2. 点击执行后按钮灰化不可重复点击
3. 执行后 ContextPanel 中显示「⏭️ 进入收入」推进按钮（且仅可点击一次推进）
4. 死亡事件展示：死者名称、派系、损失土地、损失财富，四项明确字段
5. 事件结果使用单事件视图（或仅展示第一个事件）
6. 阶段容器边距与 Senate 阶段对齐（margin ≤12）
7. 阶段内所有交互（执行→查看结果→推进）完整可操作
8. 不破坏其他阶段页面布局
9. 测试通过（全量 GUI 启动测试 + 手动验证）
