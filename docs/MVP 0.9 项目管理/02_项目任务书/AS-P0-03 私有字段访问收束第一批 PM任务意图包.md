## 二、背景说明

AS-P0-01 `PoliticalSystem`、AS-P0-02 `EconomicService` 和 AS-P0-04 广场阶段土地交易 API 化收尾已经完成，多处高风险私有字段访问随相关重构得到收束。因此，AS-P0-03 不应机械沿用早期审计结果，也不应扩展为全项目私有字段清理。

本任务是架构收口 Sprint 的最后一项 P0 主线。请 SA 先基于当前代码重新审计，识别仍然存在且会阻塞后续 P1 功能或 GUI 接入的高风险访问点，再确定“第一批”技术范围并直接向 CGT-01 发布技术任务书。

历史遗留但不具备当前 P0 风险的访问点，应登记为技术债，不要求在本轮一次性处理。

## 三、任务目标

1. 重新审计当前代码中的 UI/API 私有字段直接读写，排除已由既有任务解决的旧问题。
2. 选择一批边界明确、风险较高、可独立验收的访问点完成收束。
3. 通过 `GameState`、System、Service 或 Entity 公共方法提供受控查询、修改、清理或状态转移入口。
4. 保持现有玩法、阶段流程、自动/手动模式及多玩家信息隔离不退化。
5. 输出剩余私有字段访问清单，并区分 P0 阻塞项与 P1/P2 技术债。
6. 完成后为架构收口 Sprint 总体验收提供明确结论。

## 八、实现要求

1. SA 先输出当前代码私有字段访问审计结果，标明已解决、建议本轮处理、建议延期三类。
2. SA 为第一批选定明确对象，说明风险、公共接口归属和调用迁移边界。
3. 公共接口应表达业务意图，优先提供查询、记录、清理、状态转移等受控操作，避免直接暴露内部容器。
4. 保持依赖方向 `UI -> API -> Core/System/Service -> Entity`。
5. API 层若有调整，保持 `{success, message, data, errors}` 返回结构。
6. 自动模式和手动模式应使用一致的业务入口或等价受控入口。
7. 保持多玩家权限与信息隔离。
8. 无法在本轮安全收束的访问点应登记技术债，并说明建议优先级和触发条件。
9. 若新增或改变公共接口，应同步提出文档和函数索引更新清单。

## 九、调试日志要求

- 关键状态变更应沿用项目现有日志机制；适用时使用 `GameState.log_event()`。
- 不为纯查询方法增加噪声日志。
- 新增状态转移入口时，应保留足以定位阶段、玩家和对象的 DEBUG 信息。
- 日志不得泄漏其他玩家不可见信息。

## 十、测试要求

请 SA 根据最终选定对象制定聚焦测试，至少包括：

| 测试类型 | 要求 |
| --- | --- |
| 公共接口单元测试 | 覆盖正常操作、非法状态、重复操作或空状态 |

| 静态复查 | 确认本轮目标调用点不再直接访问对应私有字段 |

不要求为了本任务运行与修改范围完全无关的全量测试；SA 可根据影响面决定是否追加跨阶段回归。
