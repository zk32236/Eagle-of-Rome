## 二、背景说明

MVP 0.7 已具备可运行的 CLI 游戏流程，架构收口 Sprint 已完成核心系统、服务、API 和高风险私有字段访问的第一轮收束。项目后续采用“增量 GUI 接入 + 最终完整迁移”策略：

1. GUI-P0-01 建立可扩展的 GUI 通用骨架和至少一个可玩垂直切片。
2. 后续每个 P1 玩家功能同步交付最小可用 GUI。
3. GUI-P1-02 在主要 P1 功能稳定后完成残余 CLI 玩家流程迁移和产品化收尾。

GUI-P0-01 不能只是静态展示或视觉概念稿。它必须验证 GUI 能够沿 `GUI -> API -> Core/System/Service -> Entity` 方向安全驱动游戏，并为后续 P1 功能提供可复用的页面、状态、反馈、权限和测试基础。

## 六、GUI-P0-01 最低产品范围

最终范围由 SA 审查并经项目负责人确认，但不得低于以下基础：

| 能力 | 最低要求 |
| --- | --- |
| 应用启动 | 可从明确入口启动 GUI，并进入新建或加载后的游戏主界面 |
| 主游戏框架 | 具备稳定的导航、当前阶段容器、全局状态区和操作反馈区 |
| 玩家状态 | 清楚显示当前玩家、派系、回合、阶段和关键资源摘要 |
| API 适配 | GUI 通过统一适配方式调用现有 API，并处理 `{success, message, data, errors}` |
| 结构化反馈 | 统一呈现成功、失败、警告、确认、取消和加载状态 |
| 状态刷新 | 操作完成后，相关界面可从权威状态重新刷新，避免局部假状态 |
| 权限隔离 | 正确处理当前玩家、可见信息和可执行操作 |
| 阶段容器 | 后续 P1 功能可在不重建应用骨架的情况下增加页面或阶段面板 |
| 可玩切片 | 至少一个真实玩家流程可完全在 GUI 内完成，不跳回 CLI |
| 自动玩家 | 如切片涉及 AI 行动，应显示必要进度或结果并正确回到人类玩家 |
| GUI 测试 | 建立可复用的 GUI 测试框架，覆盖启动、渲染、操作、反馈与刷新 |

## 七、可玩垂直切片选择原则

SA 应审计后提出候选，不由 PM 在缺少技术审查时强行指定具体阶段。最终切片应满足：

- 使用真实 MVP 0.7 API 和状态，不使用脱离游戏的假数据演示。
- 至少包含信息展示、玩家操作、API 返回、状态刷新和流程继续。
- 能验证当前玩家与权限边界。
- 范围足够小，可独立开发和验收。
- 对后续 P1 GUI 组件具有复用价值。
- 不要求本轮迁移全部 MVP 0.7 CLI 流程。

SA 在推荐切片时，应说明为什么选择它，以及哪些能力将在后续阶段复用。

## 十、技术与架构要求

1. 保持依赖方向 `GUI -> API -> Core/System/Service -> Entity`。
2. GUI 不持有第二套权威游戏状态；显示状态应可从现有权威状态重新获取。
3. API 调用、错误映射、状态刷新和玩家权限处理应形成可复用机制。
4. 组件应服务于游戏操作效率，避免以装饰性页面取代实际工作流。
5. 固定格式区域应具有稳定尺寸和响应约束，文本不得溢出或遮挡。
6. 操作控件应使用明确、熟悉的交互形式；高影响操作应有确认。
7. 应保留 CLI 作为调试和回归入口。
8. 公共接口若有新增或调整，应登记函数索引和文档同步项。
9. GUI 框架和新增依赖必须由 SA 审查，说明维护性、测试能力、打包方式和与 Python 3.10 的兼容性。

## 十一、调试日志要求

- GUI API 调用失败、权限拒绝、状态刷新失败和阶段切换异常应有可定位日志。
- 适用时沿用 `GameState.log_event()` 和现有结构化日志方式。
- 日志应记录操作类型、阶段、当前玩家、对象标识和结果，不记录其他玩家的隐藏信息。
- 正常渲染和频繁刷新不得产生大量无价值日志。

## 十二、测试要求

SA 技术任务书至少应覆盖：

| 测试类型 | 最低要求 |
| --- | --- |
| 启动测试 | GUI 可启动并加载主框架 |
| 组件测试 | 导航、阶段容器、反馈和关键共享组件可测试 |
| API 适配测试 | 成功、失败、错误字段和异常情况正确映射 |
| 状态刷新测试 | 操作后从权威状态刷新，界面数据与 Core 一致 |
| 权限测试 | 当前玩家可操作，其他玩家不可越权或看到隐藏信息 |
| 可玩切片测试 | 至少一个成功闭环、一个非法操作、取消/返回和继续流程 |
| CLI 回归 | GUI 引入不破坏现有 CLI 关键路径 |
| 自动模式回归 | 涉及 AI 时行为与现有规则一致 |
| 视觉检查 | 在约定桌面视口下无重叠、溢出、空白主视图或关键控件不可达 |
| 全量回归 | 由 SA 根据改动影响面决定，最终报告测试数量和结果 |
