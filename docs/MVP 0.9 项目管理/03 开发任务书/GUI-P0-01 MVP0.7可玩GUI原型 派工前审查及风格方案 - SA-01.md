## Framework recommendation

### 推荐结论

采用：

```text
PySide6 + Qt Quick/QML
Python Core/API 继续作为同进程权威后端
pytest + pytest-qt 作为 GUI 测试基础
```

不采用：

- 本轮不建设浏览器 Web 应用、HTTP 服务或 WebSocket 服务。
- 不采用 Electron/Tauri + Python sidecar。
- 不采用传统 Tkinter。
- 不以 Qt Widgets 作为主要游戏界面技术。

### 推荐理由

1. 当前产品目标是 Windows 单机、本地存档和 3–6 人同机轮流操作，PySide6 可直接复用 Python Core/API，无需新增服务器和跨进程协议。
2. QML 适合阶段面板、人物列表、状态变化、地图占位和后续轻量动画，比传统 Widgets 更适合游戏化界面。
3. Python/QML 可通过 QObject、Signal、Slot 和 `QAbstractListModel` 建立明确的数据绑定层。
4. Qt for Python 提供官方部署工具，后续可以生成 Windows 可执行程序。
5. 当前解释器尚未安装 `PySide6`、`pytest-qt`、`PyInstaller`，应在 CGT-01 开发前形成受控依赖清单并验证 Python 3.10 环境。

## API readiness

### 可直接复用

| API | GUI 可用性 |
| --- | --- |
| `player_api.get_current_player()` | 可提供当前玩家基本信息 |
| `game_api.get_status_summary()` | 可提供回合、年份、国库等全局摘要 |
| `figure_api.get_figure_info()` | 已返回结构化人物数据 |
| `population_api.get_candidates()` | 已返回按官职分组的结构化候选人 |
| `population_api.campaign()` | 已包含权限和业务校验，返回统一结构 |
| `population_api.vote()` | 已包含权限、候选资格和重复投票校验 |
| `population_api.resolve_election()` | 可提供真实选举结果 |
| `forum_api` 玩家操作函数 | 多数已具有结构化返回和权限校验 |

### 必须补强

1. **GUI 安全状态快照**

   当前缺少按 `viewer_player_id` 过滤的统一 GUI DTO。`faction_api.get_factions_status()` 和 `figure_api.get_figure_info(None)` 会返回全局数据，不能直接用于多玩家 GUI。

   建议新增 UI 无关的只读查询 API，例如：

   ```text
   session_api.get_session_snapshot(state, viewer_player_id)
   population_api.get_population_view(state, viewer_player_id)
   ```

   DTO 至少包含当前玩家、派系、回合、阶段、公开国家状态、当前玩家可见人物、可执行动作和阶段进度。

2. **阶段执行边界**

   `game_api.py` 当前反向导入 CLI Command，并通过替换 `sys.stdout` 获取输出。阶段命令仍包含阻塞式 `input()` 和 CLI 循环。GUI 不得调用 `game_api.execute_phase()` 驱动交互阶段。

   GUI-P0-01 应建立薄的、UI 无关的阶段协调入口，只覆盖通用骨架和选定垂直切片，不在本轮重写全部阶段。

3. **玩家切换与隐私遮罩**

   `player_api.next_player()` 可复用，但 GUI 必须增加玩家交接遮罩。上一玩家的派系详情必须在切换成功后立即从界面模型清除，再允许下一玩家解锁。

4. **统一响应适配**

   新建 `GuiApiAdapter` 或等价类，负责：

   - 校验 `{success, message, data, errors}`。
   - 将错误、警告、成功映射为统一反馈。
   - API 完成后触发权威快照刷新。
   - 捕获未处理异常并记录日志。

5. **新游戏入口**

   GUI-P0-01 可只支持从现有 MVP 测试场景创建新游戏并进入主界面。完整存档文件管理和兼容迁移不纳入本轮。

### 不应直接复用

- `game_api.execute_phase()` / `execute_turn()` 的 CLI 命令包装。
- CLI Command 的 `input()`、`print()` 和 `_step` 内部状态。
- 直接访问 `GameState`、Faction、Figure、System 私有字段。
- 未按当前玩家过滤的全局人物或派系数据。

## Information architecture

### 页面结构

```text
启动入口
└─ 新游戏（MVP 测试场景）
   └─ GameShell
      ├─ 顶部全局状态栏
      ├─ 左侧阶段导航
      ├─ 中央阶段容器
      ├─ 右侧上下文面板
      ├─ 底部反馈/事件区
      └─ 玩家交接遮罩与确认弹窗
```

### GameShell 固定职责

| 区域 | 内容 |
| --- | --- |
| 顶部状态栏 | 回合、年份、当前阶段、当前玩家、派系、国库和关键资源 |
| 左侧阶段导航 | 七阶段顺序、当前阶段、已完成/未解锁状态 |
| 中央阶段容器 | 动态加载当前阶段或 P1 功能面板 |
| 右侧上下文 | 当前选择对象、规则摘要、费用、限制和操作结果 |
| 底部反馈区 | API 成功、错误、警告及最近事件，可折叠 |
| 遮罩层 | 玩家交接、确认、取消、加载和异常 |

建议桌面基准：

- 设计视口：`1440 x 900`
- 最低可用视口：`1280 x 720`
- 左侧导航：约 `176 px`
- 右侧上下文：约 `320 px`
- 顶部状态栏：约 `64 px`
- 底部反馈区：约 `104 px`，可折叠

### 共享状态

建议建立单一 `GuiSessionStore`：

- 内部引用唯一 `GameState`，但 QML 不直接访问 GameState。
- 对 QML 暴露只读属性和模型。
- 每次操作后重新请求权威 DTO。
- 不在 QML 中维护第二套财富、土地、投票或阶段状态。
- 提供 `currentPlayerChanged`、`phaseChanged`、`snapshotChanged`、`feedbackRaised` 等信号。

### 阶段扩展接口

阶段容器使用注册式加载：

```text
stage_id -> QML component + controller + required snapshot section
```

后续 P1 功能应能增加面板或对话框，而无需重建 GameShell。

## Playable vertical slice recommendation

### 候选 1：人口阶段“庆典 + 公职投票”闭环

**推荐采用。**

闭环：

1. 当前玩家进入人口阶段面板。
2. 查看本派系人物财富、人气和候选资格。
3. 选择人物和金额举办庆典。
4. 查看刷新后财富、人气和影响力。
5. 按官职查看候选人并投票。
6. 尝试非法金额、非当前玩家或重复投票，得到明确错误且状态不污染。
7. 完成当前玩家操作，经过玩家交接遮罩切换下一玩家。
8. AI 玩家通过现有 `AutoPlayerProcessor` 和相同 population API 行动。
9. 投票完成后调用真实选举结算，显示结果并返回阶段容器。

推荐理由：

- AS-P0-03 刚完成 population pending 和自动/手动统一 API，技术边界最干净。
- 同时验证数字输入、人物选择、候选列表、权限、多玩家交接、自动玩家、结果反馈和状态刷新。
- 人物卡、属性行、资源输入、选择列表、确认弹窗和结果面板可直接复用于人物属性、经验、忠诚、腐败、审判和收买等 P1 GUI。
- 比元老院完整迁移风险低，比单一土地交易更能验证通用骨架。

需要的最小 API 补强：

- 当前玩家可见的人口阶段 DTO。
- 当前人口阶段进度和已投票状态。
- GUI 可调用的当前玩家完成/切换协调入口。
- 选举结果结构化数据可适当补强为 `{office, figure_id, name, faction_id}`。

### 候选 2：广场阶段私地交易

优点：

- API 边界已完成 AS-P0-04 收束。
- 表单、确认、权限、pending 和结算容易验证。

不足：

- 流程过窄，难以验证通用阶段容器、AI 行动和复杂状态刷新。
- 对后续政治/人物 P1 GUI 的组件复用弱于人口阶段。

### 候选 3：元老院单提案投票

优点：

- 最能展示项目核心特色。

不足：

- 元老院步骤、提案类型、否决、自动投票和执行链较复杂。
- 容易扩大为整个元老院阶段迁移，不适合作为第一个原型。

结论：GUI-P0-01 采用候选 1；候选 2 可作为后续第二个低风险切片；候选 3 延至后续增量 GUI 或 GUI-P1-02。

## GUI testing plan

### 工具

- `pytest`
- `pytest-qt`
- `QT_QPA_PLATFORM=offscreen` 的无头组件测试
- Qt 截图抓取和像素非空检查
- SA 在桌面模式下执行可见窗口手工验证

### 必须覆盖

1. GUI 启动并渲染 GameShell。
2. `GuiApiAdapter` 正确映射成功、失败、errors 和异常。
3. 权威快照刷新后人物财富、人气、影响力和投票状态一致。
4. 当前玩家可操作；非当前玩家被拒绝。
5. 玩家交接时旧玩家敏感数据立即清除。
6. 庆典成功、资金不足失败、取消不提交。
7. 投票成功、重复投票失败、非法候选失败。
8. AI 操作后返回人类玩家且界面刷新。
9. 选举结果真实来自 API，不使用假数据。
10. `1440 x 900` 与 `1280 x 720` 截图无重叠、溢出、空白主视图或不可达控件。
11. 现有 CLI population、API 和全量 pytest 回归通过。

## Risks and dependencies

1. 当前未安装 PySide6、pytest-qt 和 GUI 打包依赖，需要项目负责人在开发任务发布后批准安装。
2. `game_api -> CLI Command` 的反向依赖不能作为 GUI 主入口；本轮只补选定切片与通用壳层所需接口。
3. 当前全局人物/派系查询可能泄漏其他玩家信息，必须先建立 viewer-scoped DTO。
4. QML 不能持有或修改 GameState 私有状态。
5. 本地多人玩家交接必须先清除旧数据，再显示解锁遮罩。
6. UI 线程不得执行可能阻塞的 CLI 输入循环。
7. 中文字体和图标资产必须随应用明确管理，不能依赖开发机偶然存在的字体。
8. 原型阶段仍需控制素材许可和包体积。
