## Selected first-batch targets

### 1. 人口阶段 pending 状态

当前问题：

- `population_api.py` 直接读取、追加 `_population_pending["campaigns"]` 和 `_population_pending["votes"]`。
- `auto_player_processor.py` 在 bypass 路径直接修改人物与 pending，造成自动/手动入口分叉。
- `phase_population.py` 直接读取并清空 pending。

公共接口归属：

- `GameState` 提供人口庆典记录、投票查询/记录或替换、快照、清理方法。
- `population_api.campaign()` / `vote()` 增加显式 `bypass_permission=False` 参数，自动模式仍经 API 执行业务规则。
- `resolve_election()` 和命令层只读取公共快照；命令层通过公共方法清理。

### 2. WarSystem 跨层容器访问

当前问题：

- `phase_forum.py` 直接向 `_active_wars` 追加起义战争。
- `naval_system.py` 直接读取 `_threats`。
- `forum_api.py`、`phase_forum.py`、`auto_player_processor.py`、`phase_population.py` 直接遍历 `_war_discard`。
- `phase_population.py` 直接读取并清空 `_legions_to_disband`，虽然 `clear_legions_to_disband()` 已存在。

公共接口归属：

- `WarSystem` 增加或补强只读查询：已解决战争、海战威胁。
- `WarSystem` 增加起义战争登记入口，负责状态和容器一致性。
- 人口阶段使用 `clear_legions_to_disband()` 一次性取出并清理队列；失败时仅将未成功处理的编号重新加入。

### 3. 总督交接状态转移

当前问题：

- `phase_resolution.py` 直接读取和清理 `_old_governor_id`、`_governor_designate_id`，并直接设置 `_governor_id`。

公共接口归属：

- `Province` 提供一个表达“完成候任总督交接”的原子状态转移方法，返回旧总督 ID 和新总督 ID。
- `phase_resolution.py` 保留人物状态更新、输出和日志，但不得直接修改 Province 私有字段。
- 无有效候任人时也必须清理本轮候任/旧总督临时状态，保持旧行为。

### 4. 公地认购的 Figure 私地写入

当前问题：

- `forum_api.resolve_forum()` 公地认购结算直接执行 `figure._land_private += actual_buy`。

公共接口归属：

- 复用 `Figure.buy_land(amount, price_per_unit)` 完成人物财富与私地的原子变更。
- API 继续负责国库、公地配额和结算结果；不得改变价格、排序和配额规则。

## Architecture risks

- `population_api` 当前 bypass 由全局测试配置控制。新增显式 bypass 参数时必须保持默认权限校验，且只允许自动玩家编排层传入。
- 待解散军团队列若“先清空后处理”，必须在处理失败时恢复失败项，避免状态丢失。
- 总督交接方法应只管理 Province 状态；人物 `office`、`is_absent`、影响力和玩家可见输出仍由阶段编排层处理。
- 战争查询必须返回副本，战争登记必须防止同一战争重复进入多个容器。
- 公地认购改用 `Figure.buy_land()` 时不得重复扣款。
