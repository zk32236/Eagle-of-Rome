# EOR GUI Visual Diff Report — Phase 1（天命阶段）

**对比对象**

- **Actual：** `GUI acutal phase 1 - 4.PNG`
- **Target：** `GUI design - Phase 1.PNG`
- **用途：** 供 AI 开发 Agent 根据差异进行定向微调
- **审查范围：** 仅比较截图中可见的布局、层级、视觉状态与控件排列；不评价未展示的代码逻辑

---

## 1. 总体结论

当前实现已经复现了目标原型的整体框架：

- 顶栏
- 左侧阶段导航
- 中央阶段区
- 右侧状态与日志栏
- 底部查询栏
- 深墨外壳、朱红 HUD、象牙白中央桌面

但当前版本尚未达到视觉验收标准，主要原因不是整体样式，而是**中央阶段内容的布局与可见性发生严重偏差**。目标原型中的阶段编号、标题、说明、步骤条和信息卡应集中显示在中央区顶部；实际实现中，大部分内容被移位到中央区域中下部并呈极低对比度，几乎不可读。

因此，本轮整改应优先修复中央阶段内容容器，不应继续做全局配色或布局重构。

## 3. 必须优先处理的根因

### 3.1 中央内容可能错误继承禁用态

实际截图中，阶段标题、步骤条和部分正文均呈统一的低透明度。这不像单个颜色设置错误，更像以下情况之一：

- 父容器被设置为 `opacity < 1`
- 当前阶段误套用了 `.disabled`、`.locked`、`.inactive` 或等价状态
- QML 控件的 `enabled: false` 导致整棵子控件自动降透明度
- 页面切换动画未恢复最终 opacity
- 透明遮罩层覆盖在内容上方
- 文本颜色使用了适用于深色背景的浅色值，但实际容器为象牙白
- 活动页面与隐藏页面同时存在，错误页面叠加在当前页面上

AI 开发 Agent 应先定位这一根因，而不是逐个修改文字颜色。

### 3.2 中央容器的垂直布局策略错误

实际元素分布显示，中央容器很可能存在以下布局问题：

- 使用了 `justify-content: space-between`
- QML `ColumnLayout` 中存在错误的 `Layout.fillHeight`
- 中间 spacer/Item 吞掉剩余高度
- 标题区被 anchor 到错误位置
- 内容顺序与 DOM/QML 层级不一致
- 信息卡被单独放入顶部 overlay，而标题区仍在主布局中

目标结构应是：

```text
中央阶段桌面
├─ 阶段编号 1/7
├─ 阶段标题
├─ 阶段说明
├─ 步骤条
├─ 操作说明卡
├─ 其余阶段内容 / 留白
└─ 主操作按钮（底部居中）
```

## 4. 推荐整改顺序

### 第一批：阻断性修复

1. 修复中央内容透明度/禁用态继承。
2. 恢复阶段标题区的顶部流式布局。
3. 将步骤条和信息卡恢复到正确顺序。
4. 修复玩家区横向截断。
5. 核对进度字段 `4/4` 与目标 `2/2` 的业务映射。

### 第二批：结构一致性

1. 恢复底部查询按钮顺序。
2. 补齐顶栏稳定度和战争数值。
3. 缩减主操作按钮宽度。
4. 压缩右侧日志区冗余标题与间距。

### 第三批：视觉微调

1. 微调左侧阶段栏内边距。
2. 调整品牌区宽度。
3. 校正底栏图标和文字基线。
4. 规范正式验收截图，移除开发控制台。

---

## 5. 验收标准

整改完成后，Phase 1 截图至少应满足：

- 阶段编号、标题、说明、步骤条和信息卡在中央区顶部完整可见。
- 正常可操作阶段不存在大面积半透明或禁用视觉。
- 主按钮位于底部中央，尺寸与目标接近。
- 右侧玩家信息不截断。
- 右侧进度与真实阶段流程一致。
- 顶栏所有已绑定状态字段有明确数值。
- 底部 12 个查询按钮顺序与批准原型一致。
- 截图不包含终端、控制台或调试文字。
- 不改变已批准的五区布局和整体材质体系。

---

## 6. 给 AI 开发 Agent 的执行边界

本轮属于**视觉一致性修复**，不得借机：

- 重构五区布局
- 更改阶段流程
- 改变功能语义
- 重排查询入口（除恢复批准顺序外）
- 新增未经批准的卡片或说明
- 用新的配色主题替代现有深墨、朱红、象牙白和羊皮纸体系
- 通过隐藏真实内容来达到截图相似

应优先通过修复状态绑定、布局容器、padding、opacity、文字颜色和控件顺序完成整改。
