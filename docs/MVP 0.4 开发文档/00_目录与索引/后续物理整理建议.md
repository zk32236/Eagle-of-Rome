# 后续物理整理建议

当前已完成非破坏性索引整理。若后续希望真正移动或重命名文件，建议分两步执行。

## 第一阶段：只整理顶层散文件

建议新增目录：

- `01_产品策划与规则资料`
- `02_架构设计与技术规范`
- `05_UI与视觉资料`
- `99_历史归档`

把顶层散文件按主题移动进去。该阶段风险较低，因为不触碰已有大目录内部结构。

## 第二阶段：整理大目录内部

建议对 `MVP 046功能基线文档` 单独处理，因为它同时包含：

- 项目输入包
- 策划师输出
- 架构师输出
- 开发测试输出
- UI 设计师输出
- MVP 0.5 与 MVP 0.7 历史开发输出

该目录更像“0.4.6 到 0.7 的协作全量档案”，不建议简单归入 MVP 0.4。

## 不建议立即执行的动作

- 不建议批量重命名 300+ 个 Word 文档。
- 不建议把历史 `.py` 文件混入当前源码目录。
- 不建议删除“过时”文档；这些材料对理解早期决策仍有价值。
- 不建议把 MVP 0.7 正式基线文档继续留在历史目录中作为唯一副本，应以当前正式目录为准。
